# Known Limitations

This document records honest, unresolved limitations of SentinelOps.
It is intended for external reviewers, evaluators, and potential adopters.
Nothing here is hidden or minimized.

---

## 1. Simulation-Only Validation

**All evaluation is performed on synthetic incident datasets.**

No production telemetry has been used. Benchmark scores reflect performance
on simulated incidents generated from probabilistic templates and chaos-injected
replays. Real-world incident patterns may differ substantially.

**Implication:** Claimed recall, precision, and F1 scores cannot be extrapolated
to production without validation on real incident data.

---

## 2. LLM Dependency

**Core reasoning depends on an external LLM API** (OpenAI or Anthropic).

- LLM outputs are non-deterministic even with fixed seeds (sampling-based generation).
- API availability is not guaranteed.
- LLM behavior changes across model versions without notice.
- Prompt injection via log content is a recognized and only partially mitigated risk.

**Implication:** Reproducibility guarantees apply to the evaluation scaffolding,
not to LLM-generated reasoning outputs. Two runs with identical inputs may
produce different explanations.

---

## 3. No External Security Audit

No third-party security audit has been performed.

Known unreviewed surfaces:
- FastAPI route authentication
- LangGraph checkpoint serialization
- Operator input deserialization
- Telemetry ingestion pipeline

---

## 4. Architectural Coupling: evaluation → runtime

`apps/api-server/src/evaluation/replay_integration.py` imports from
`runtime.continuous_evaluator`, `runtime.drift_monitor`, and
`runtime.operational_regression`.

This violates the intended layer boundary (evaluation should not depend on runtime).
The coupling exists from Phase 48 integration work and has not been refactored.

**Impact:** Changes to runtime modules can silently break evaluation pipelines.
The architecture validator flags this coupling explicitly.

---

## 5. Confidence Calibration Is Uncalibrated

Confidence scores are produced by heuristic combination of evidence signals —
they are **not calibrated against empirical outcomes**.

A confidence of 0.80 does not mean the system is correct 80% of the time.
Calibration curves have not been measured on holdout sets.

---

## 6. Trust Model Is Bounded Simulation

The trust model in `operators/workflow/` simulates operator trust decay and
recovery. The decay/gain rates (lose 2–3x faster than gain) are theoretically
motivated but not empirically validated against real operator behavior.

---

## 7. Autonomy Is Not Permitted

The deployment readiness validator classifies this system as `experimental`
or `staging_capable` against the current evidence base.

**Autonomous operation is prohibited** in all configurations.
All remediation recommendations require human review before execution.

---

## 8. No Durable Checkpoint Persistence

LangGraph checkpointing uses `MemorySaver` (in-process only).
Cross-process recovery from interrupted workflows is not supported.
`langgraph-checkpoint-postgres` integration is not implemented.

---

## 9. Benchmark Contamination Risk in Legacy Datasets

The operational chaos dataset (`simulation/datasets/operational_chaos/incidents.json`)
was generated by the same codebase that evaluates it. While fingerprinting and
anti-contamination guards are in place, the dataset was not generated by an
independent process. This limits claims of generalization.

---

## 10. Observability Coverage Is Partial

The observability coverage auditor reports that not all source modules contain
logging or metrics instrumentation. Modules in `evaluation/`, `causality/`, and
`semantics/` layers have partial coverage.

This means failures in those modules may produce no observable signal in
production monitoring.

---

## Summary

| Limitation | Severity | Workaround |
|---|---|---|
| Simulation-only validation | High | Validate against real incidents before production |
| LLM non-determinism | High | Run multiple seeds, report variance |
| No security audit | Medium | Conduct audit before external exposure |
| eval→runtime coupling | Medium | Refactor replay_integration.py |
| Uncalibrated confidence | Medium | Collect calibration data in shadow mode |
| Unvalidated trust rates | Low | A/B test with real operator cohorts |
| No durable checkpointing | Medium | Add postgres checkpoint adapter |
| Partial observability coverage | Low | Add structured logging to uncovered modules |
