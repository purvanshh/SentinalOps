# Why Automated Root-Cause Analysis Is Harder Than It Looks

**Lessons from Building SentinelOps AI**

Author: Purvansh Sahu, 3rd-year CS, Scaler + BITS Pilani  
Date: 2026-05-25

---

## Abstract

We built an uncertainty-aware incident reasoning system with production-grade infrastructure — LangGraph orchestration, multi-layer provider resilience, deterministic evaluation, and 2,139 passing tests — and achieved 0.9917 router accuracy, 1.0000 grounding, and 0.7521 safety. Yet root-cause accuracy on our 121-incident benchmark remains **0.1502 (lexical) / 0.2714 (semantic)**. This note explains why that gap persists, what we tried to close it, and what we learned about the fundamental difficulty of automated causal inference from operational telemetry.

---

## 1. The Problem Is Not the Infrastructure

Most engineering effort in AI-assisted operations goes into orchestration, safety, and evaluation integrity. We did that work:

- **Provider resilience:** 4-layer fallback chain (primary → secondary → local Ollama → deterministic classifier) ensures the pipeline never halts.
- **Execution safety:** JWT approval tokens, tool allowlists, risk-tier classification, and operating mode management gate all dangerous actions.
- **Evaluation integrity:** Golden labels are structurally isolated from agent cognition. Runtime assertions (`_assert_no_golden_contamination()`) verify that benchmark answers never leak into mocked tool responses.
- **Uncertainty honesty:** The `UncertaintyCollapseGuard` prevents overconfident attribution, and the `AmbiguityResolver` refuses attribution entirely when evidence is insufficient.

That infrastructure is necessary but not sufficient. The hard part is **causal inference from sparse, symptom-biased telemetry**.

Production incident data tells you *what* broke — latency spike, connection timeout, OOM kill. It rarely tells you *why* — misconfiguration, upstream degradation, connection pool leak after a deployment. A human SRE bridges this gap with institutional knowledge: "DNS timeout at this scale usually means resolver misconfiguration," or "connection pool exhaustion after a deployment usually means the new version leaks connections."

Our benchmark golden labels encode this causal knowledge. Our mocked evidence does not. That mismatch is the fundamental bottleneck.

## 2. The Pipeline: What Actually Happens

The root-cause analysis pipeline in SentinelOps is a 5-stage deterministic process (LLM synthesis is optional and only used in production):

```
evidence_normalizer.py    →  Flatten agent outputs into typed evidence items
evidence_builder.py       →  Construct timestamped event timeline
causal_graph.py           →  Generate candidate causes via keyword matching
probabilistic_reasoner.py →  Score candidates with weighted softmax
uncertainty.py            →  Assess confidence, detect contradictions, decide escalation
```

### 2.1 Candidate Generation (`causal_graph.py`)

Candidates are generated by pattern-matching on evidence content:

| Evidence Signal | Generated Candidate | Match Score |
|----------------|-------------------|-------------|
| `dns_resolution_ms` metric anomaly | DNS resolution failure | 0.88 |
| `ConnectionPoolExhaustedException` in logs | Connection pool exhaustion | 0.93 |
| Deployment `v2.3.1` detected | Deployment regression | 0.78 |
| `external_api_latency` metric | External dependency degradation | 0.87 |
| No patterns match | Unknown service degradation (fallback) | 0.20 |

Topology propagation adds cascade hypotheses for upstream/downstream services when a candidate scores ≥ 0.6.

**The limitation:** This is keyword matching, not causal reasoning. The system can identify *what kind of failure* the evidence describes, but cannot infer *why* that failure occurred. `dns_resolution_ms` anomaly → "DNS resolution failure" is a symptom restatement, not a root cause.

### 2.2 Probabilistic Scoring (`probabilistic_reasoner.py`)

Candidates are scored with a weighted formula:

```python
raw_score = (
    confidence * 0.55 +
    prior_probability * 0.20 +
    temporal_score * 0.15 +
    evidence_coverage * 0.10
) * support_strength
```

Probabilities are distributed via temperature-scaled softmax (`temperature=1.0`), then the `UncertaintyEngine` applies sparse-evidence penalties and contradiction adjustments.

**The limitation:** These weights are hand-tuned. No learning mechanism exists to improve them from feedback. The scoring correctly ranks candidates relative to each other, but cannot elevate symptom-level descriptions into cause-level language.

### 2.3 Uncertainty Assessment (`uncertainty.py`)

The engine aggregates multiple uncertainty sources:

- **Sparse evidence penalty:** 1 item → 0.6× multiplier, 2 items → 0.8×
- **Missing telemetry:** −0.15 per missing type (metrics, logs, deployment)
- **Contradiction detection:** Temporal contradictions (severity 0.35), signal conflicts (severity 0.22)
- **Hypothesis stability:** Gap between top-2 probabilities

State resolution follows a priority cascade:

| Condition | State | Action |
|-----------|-------|--------|
| ≥2 missing telemetry types | `insufficient_telemetry` | Escalate |
| Any contradictions | `conflicting_signals` | Escalate |
| Confidence < 0.35 AND stability < 0.1 | `unknown_cause` | Escalate |
| Confidence < 0.55 | `low_confidence_escalation` | Escalate |
| Otherwise | `stable` | Proceed |

**The limitation:** Better calibration makes the system's confidence *meaningful* — it no longer claims certainty it doesn't have. But honest uncertainty is not the same as correct answers. The system now says "I'm 18% confident this is DNS failure" instead of "I'm 6% confident this is DNS failure." Both are still wrong when the actual cause is "DNS resolver misconfiguration."

## 3. The Measurement: What the Numbers Mean

### 3.1 Benchmark Structure

Our benchmark (`benchmark_suite_v1.json`) contains 121 incidents across 28 categories. Each incident provides:

- **Alert payload:** What triggered the incident (title, severity, service, metrics snapshot)
- **Mocked tool responses:** What the evidence agents "see" (metric anomalies, log errors, deployment events)
- **Golden labels:** What a human SRE would conclude (root cause, remediation, blast radius)

Example of the gap:

| Component | Content |
|-----------|---------|
| **Mocked evidence** | `dns_resolution_ms` observed 450ms (expected 5-50ms), z_score=8.2 |
| **Pipeline output** | "DNS resolution failure in payment-api" |
| **Golden label** | "DNS resolver misconfiguration causing resolution timeouts" |

The pipeline correctly identifies the *symptom domain* (DNS) but cannot infer the *mechanism* (misconfiguration) because that information isn't in the evidence.

### 3.2 Scoring Methodology

Two complementary metrics:

- **Lexical (F1 token overlap):** Measures exact vocabulary match. "DNS resolution failure" vs "DNS resolver misconfiguration" shares only "DNS" and "resolution" — penalized heavily.
- **Semantic (MiniLM cosine similarity):** Measures meaning alignment. Rescaled from raw cosine: `(cosine - 0.3) / 0.7`. Partially compensates for vocabulary mismatch.

### 3.3 Current Results

| Metric | Value | Interpretation |
|--------|-------|----------------|
| Root-cause accuracy (lexical) | 0.1502 | Low token overlap — expected given symptom-cause vocabulary gap |
| Root-cause accuracy (semantic) | 0.2714 | Moderate meaning alignment — pipeline identifies correct domain |
| Mean cosine similarity | 0.4712 | Raw embedding distance shows partial semantic capture |
| Grounding score | 1.0000 | All citations reference valid evidence — no hallucination |
| Hallucination score | 1.0000 | No causal claims contradict available evidence |
| Workflow completion | 1.0000 | All 121 incidents processed end-to-end |
| Safety score | 0.7521 | Remediation risk-tier enforcement working |
| Router accuracy | 0.9917 | Classification is solved (but easier than RCA) |

The system is *grounded* (never hallucinates) and *safe* (never over-attributes), but not *accurate* in the sense of matching human causal reasoning.

## 4. What We Tried

### 4.1 Evidence-Grounded Candidate Generation

**Hypothesis:** Generating candidates directly from evidence specificity (metric names, log signatures, deployment versions) instead of generic pattern hints would improve accuracy.

**Implementation:** Rewrote `causal_graph.py` to map specific signals to specific candidates:
- `dns_resolution_ms` → DNS-specific candidate
- `ConnectionPoolExhaustedException` → pool-exhaustion candidate  
- Deployment version detected → regression candidate

**Result:** Candidate specificity jumped from ~10% to 42%. Root-cause accuracy moved from 0.0820 to 0.1447 (lexical). The pipeline now produces "Database connection pool exhaustion after deployment v2.3.1" instead of "Unknown service degradation."

**Why it wasn't enough:** The candidates are symptom-specific but the benchmark expects cause-specific vocabulary. "DNS resolution failure" and "DNS resolver misconfiguration" describe the same incident but share few tokens.

### 4.2 Constrained LLM Synthesis

**Hypothesis:** A constrained LLM prompt can perform symptom-to-cause translation using general operational knowledge, without benchmark contamination.

**Implementation:** Added `synthesize_root_cause_hypothesis()` — a post-selection step that takes the winning candidate and evidence context, asks the LLM for a ≤12-word cause hypothesis. Rejects vague outputs ("service degradation", "unknown"). Falls back to candidate title.

**Production result (Llama 3.1 70B):** Excellent. Manual spot checks:
- BM-006: "External payment processor degraded, adding latency to checkout"
- BM-010: "DNS resolver misconfiguration causing resolution timeouts"
- BM-013: "Database connection pool exhaustion after deployment v2.3.1"

**Evaluation challenge:** We attempted local Ollama models for deterministic measurement:

| Model | Lexical | Semantic | Mean Cosine | Verdict |
|-------|---------|----------|-------------|---------|
| Deterministic mock | 0.1502 | 0.2714 | 0.4712 | **Retained** |
| `gemma3:1b` | 0.0710 | 0.1257 | 0.3485 | Rejected — too weak |
| `qwen2.5:7b` | 0.1689 | 0.2437 | 0.4246 | Rejected — semantic regression |

Small models collapse into generic output or hallucinate unsupported mechanisms. The handcrafted `DeterministicEvalSynthesisClient` preserves candidate specificity better at small scale.

**Decision:** Retain deterministic mock for evaluation. Accept that the benchmark under-measures production capability.

### 4.3 Calibration and Uncertainty Tuning

**Hypothesis:** Systematic underconfidence (87.5% rate, ECE 0.2045) was destroying signal from good candidates.

**Implementation:**
- Reduced softmax temperature from 1.35 to 1.0
- Replaced cliff penalty (0.1× for <3 evidence items) with tiered penalty (0.6× for 1 item, 0.8× for 2)
- Fixed `UncertaintyEngine.assess()` to use `max(probabilities)` instead of `probabilities[0]`

**Result:**

| Metric | Before | After |
|--------|--------|-------|
| Underconfidence rate | 87.5% | 0.0% |
| ECE | 0.2045 | 0.0897 |
| Avg confidence (incorrect) | 0.06 | 0.1834 |
| Calibration grade | POOR | FAIR |

**Why it wasn't enough:** Better calibration makes the system trustworthy but not correct. Trustworthy wrong answers are better than confident wrong answers, but they're still wrong answers.

## 5. Why the Problem Is Fundamentally Hard

Three structural issues make automated root-cause analysis harder than standard classification or retrieval:

### 5.1 The Symptom-Cause Asymmetry

Metrics and logs describe symptoms. Root causes are inferred. The inference requires causal models that are not present in the telemetry itself.

```
What telemetry provides:     dns_resolution_ms = 450ms (z_score = 8.2)
What inference requires:     Knowledge that high DNS latency + no upstream change
                             usually indicates resolver misconfiguration
What the benchmark expects:  "DNS resolver misconfiguration causing resolution timeouts"
```

This is not a retrieval problem (the answer isn't in a document). It's not a classification problem (the categories aren't predefined). It's an *abductive reasoning* problem — inferring the most likely explanation from incomplete observations. Abductive reasoning requires domain knowledge that isn't in the evidence.

### 5.2 The Evaluation Contamination Trap

It is trivial to improve benchmark scores by leaking golden-label vocabulary into the agent. We caught ourselves doing this and added runtime assertions:

```python
def _assert_no_golden_contamination(incident_fixture, mocked_responses):
    """Raises ValueError if golden labels appear in mocked tool responses."""
```

Any "fix" that involves tuning candidate titles or synthesis prompts to match benchmark words is contamination, not progress. The `AmbiguityResolver` and `UncertaintyCollapseGuard` exist partly to prevent the system from claiming knowledge it doesn't have — even when that would improve scores.

The five causal reality states enforce this discipline:

| State | Confidence Cap | System Behavior |
|-------|---------------|-----------------|
| `stable_cause` | 0.95 | Proceed with attribution |
| `competing_causes` | 0.60 | Surface alternatives to operator |
| `insufficient_evidence` | 0.35 | **Refuse attribution entirely** |
| `temporally_unstable` | 0.50 | Flag timestamp sensitivity |
| `observation_conflict` | 0.40 | **Refuse attribution entirely** |

The system prefers "uncertain but honest" over "confident but wrong." This is the correct engineering choice, but it means benchmark scores stay low when evidence is genuinely insufficient.

### 5.3 The Determinism-Capability Tradeoff

Deterministic evaluation requires pinned models, fixed seeds, and cached responses. But symptom-to-cause inference is exactly the kind of nuanced language task where model capability matters.

Our production model (Llama 3.1 70B) handles it well. Our benchmark cannot run a 70B model locally with deterministic guarantees. So we measure a weaker proxy — and the proxy correctly shows that small models can't do abductive reasoning reliably.

This creates an honest but frustrating situation: the system works better than we can prove in automated evaluation.

### 5.4 The Keyword Matching Ceiling

The current `causal_graph.py` uses hardcoded keyword vocabularies:

```python
# From causal_graph.py — metric pattern matching
if "dns" in metric_lower or "resolution" in metric_lower:
    return CandidateCause(title="DNS resolution failure", score=0.88, ...)
if "pool" in metric_lower or "connection" in metric_lower:
    return CandidateCause(title="Connection pool exhaustion", score=0.90, ...)
```

This works for known failure modes but cannot generalize to novel incidents. A new failure pattern (say, "gRPC deadline exceeded due to TLS certificate rotation") won't match any existing keyword set and will fall through to the generic "Unknown service degradation" fallback at score 0.20.

The system has exactly 8 hardcoded service markers, ~12 metric patterns, ~8 log patterns, and 1 deployment pattern. Real infrastructure has thousands of failure modes.

### 5.5 The Evidence Richness Gap

Our mocked evidence provides:
- Metric anomalies (name, observed value, expected range, z-score)
- Log error signatures (message, count, trace IDs)
- Deployment events (version, timestamp, commit summary)

Real production telemetry would also provide:
- **Config change logs:** "TTL reduced from 300s to 5s"
- **Dependency status:** "Stripe API status: degraded"
- **Deployment diffs:** "connection pool size changed from 100 to 10"
- **Runbook correlations:** "Last 3 DNS incidents were resolver misconfigs"
- **Topology changes:** "New upstream added 2 hours before incident"

These causal hints are what human SREs use to bridge the symptom-cause gap. Without them, even a perfect algorithm can only restate symptoms.

## 6. What We Are Doing Next

### 6.1 Enrich Mock Evidence with Causal Hints

Instead of fighting the symptom-cause gap at the synthesis layer, close it at the evidence layer. Mock evidence will include config changes, dependency status, and deployment diffs — signals that real production telemetry would contain.

**Expected impact:** +0.10 to +0.20 semantic accuracy without touching the synthesis prompt.

### 6.2 Learned Candidate Generation

Replace the rule-based `causal_graph.py` with a lightweight classifier trained on incident-evidence-cause triples. Input: flattened evidence features. Output: candidate cause labels.

This is safe because the classifier learns from *evidence features*, not from golden label text. It generalizes "DNS timeout + high latency" → "DNS infrastructure issue" without memorizing benchmark IDs.

**Expected impact:** +0.15 to +0.25 semantic accuracy. Enables handling of novel incidents.

### 6.3 Production Validation Pipeline

Benchmarks measure proxies. The real test is operator adoption:
- Deploy with Llama 3.1 70B synthesis
- Capture operator override decisions (agree / disagree / modify)
- Feed override signals back into retrieval and calibration
- Report operator agreement rate as the primary success metric

**Target:** 60% operator agreement on root-cause hypotheses within 3 months of deployment.

## 7. What We Learned

1. **Evaluation integrity is harder than building the system.** The Phase 40 contamination fix taught us that benchmarks can lie silently. Runtime assertions against golden-label leakage are non-negotiable.

2. **Determinism enables debugging but limits measurement.** A deterministic benchmark proves the pipeline is not broken. It cannot prove the pipeline is good. That requires production validation.

3. **Small models fail at constrained reasoning.** We assumed `qwen2.5:7b` would follow our synthesis prompt as well as a 70B model. It does not. Instruction following scales with model size more than we expected.

4. **Calibration honesty is a prerequisite, not a solution.** Fixing underconfidence made the system trustworthy. It did not make the system correct.

5. **Safety and accuracy are in tension.** The `UncertaintyCollapseGuard` and `AmbiguityResolver` correctly refuse attribution when evidence is insufficient. This is the right engineering choice — but it means the system will never achieve high benchmark scores on incidents where evidence is genuinely sparse.

6. **The symptom-cause gap is not a bug — it's the core research problem.** No amount of pipeline engineering closes a gap that requires domain knowledge not present in the input data. The path forward is richer evidence, not cleverer scoring.

---

## 8. Conclusion

SentinelOps AI is a structurally sound incident reasoning system with honest evaluation, robust safety, and a solved orchestration layer. Its root-cause accuracy is limited not by architecture but by the fundamental difficulty of causal inference from symptom-biased telemetry.

The system does what it can prove: it identifies the correct failure domain, grounds all claims in evidence, refuses attribution when evidence is insufficient, and escalates to operators when uncertain. What it cannot yet do — and what no current system does reliably — is bridge the gap between "what broke" and "why it broke" without the causal context that human SREs carry in their heads.

The benchmark numbers will improve when the evidence supports it, not when we tune against it.
